# 與 Readdy 差異化分析

更新日期：2026-03-14

## 評估前提
目前我們在思考的不是「Readdy 能不能做網站」，而是：
- 如果 Readdy 未來把 CMS、Visual Editing、部署流程都補齊
- 我們自己還有沒有必要做「下載 code 後接 CMS、部署、半自動化上架」這一段
- 我們真正該投入的差異化能力到底是什麼

## 核心判斷
如果 Readdy 真的提供成熟的 CMS、Visual Editing、部署與內容發布流程，  
那麼我們原本規劃中的這一段：

- Readdy 匯出 code
- 串接 Sanity
- 接 Visual Editing
- 部署到 Vercel
- 做 prerender / 發布流程

大部分都會被 Readdy 平台能力覆蓋。

換句話說，若 Readdy 走到這一步，我們就不需要再重複投入同一段「網站內容管理與發布」基礎設施。

## Readdy 補齊 CMS 後，仍不一定會覆蓋的部分
即使 Readdy 補齊網站 CMS，它也不一定會覆蓋我們想做的「產品平台層」。

還可能需要由我們自己掌握的部分：
- 與 CRM / OMO / 行銷工具的深度整合
- 我們自己的客戶管理、專案管理與權限流程
- 多客戶、多站點、多品牌的產品化治理
- 廣告追蹤、名單歸因、到店/成交資料回流
- 與內部系統或第三方醫療營運流程的串接

所以如果 Readdy 把 CMS 做完整，它會取代的是「網站內容層」，  
但不一定會取代「網站與營運系統之間的整合層」。

## 網站與 CRM / OMO 需要整合的項目
目前可明確想到的整合需求如下。

### 流量追蹤層
- GA4
- Meta Pixel
- Conversion API
- Google Search Console
- Bing Webmaster Tools
- Google Ads / Meta 廣告事件與驗證

### 名單與身分層
- 表單送出進 CRM
- 預約點擊、電話點擊、LINE 加好友進 CRM
- 保存 UTM / gclid / fbclid 等來源參數
- 同一位名單的去重與歸戶

### 歸因與 OMO 層
- 從網站事件追到實際預約
- 從預約追到到店 / 到診 / 成交
- 線下結果回傳廣告平台，形成完整轉換閉環

### 合規與治理層
- Cookie / consent 管理
- 個資與敏感資料保護
- 權限分工與審計

## 設計前置流程也值得做差異化
除了網站生成與後續營運整合，設計前期的資料蒐集與整理本身也很值得產品化。

目前設計師常需要花時間：
- 讀品牌需求表單
- 到官網、FB、IG、Google 商家、LINE 補資料
- 交叉驗證資訊
- 重新整理成可以討論的品牌 brief

這一段若能做成：
- 精簡版客戶 intake
- 公開資料自動抓取與摘要
- 設計師確認與補充

就能直接節省大量前期工時，也會讓後續網站生成、CMS 初始內容、追蹤設定與 CRM 建檔更順。

更完整的整理請見：
- [設計前置與問卷調整建議](./設計前置與問卷調整建議.md)

## 如果 Readdy 做了 CMS，我們的差異化應放在哪裡
若 Readdy 已經能解決網站生成、內容編輯、發布與部署，  
我們更值得投入的不是再做一個類似 CMS，而是做：

- 行銷追蹤與事件治理
- 與 CRM 的名單整合
- 線上到線下的轉換歸因
- 多客戶營運流程
- 客戶成功與後台治理

也就是說，我們真正的差異化比較可能是：
「讓網站成為營運與轉換系統的一部分」，  
而不是只是一個能被編輯和上線的網站。

## 關於登入與客戶使用體驗
若採用 Sanity 這類第三方 CMS / Visual Editing：
- 客戶若要直接進 Studio 編輯，就需要有 Sanity 身分
- 最穩做法是由我們邀請客戶成為該專案的 Editor
- 不建議自動幫客戶生成一組 Sanity 帳密來代管登入

這代表：
- 如果我們接受客戶使用第三方登入體驗，Sanity 原生模式可以很快上線
- 如果我們要完整品牌化、統一登入入口，長期仍需要自己做前台編輯介面與 BFF

## 現階段建議
### 如果 Readdy 尚未補齊 CMS
- 可用 Readdy + Sanity + Vercel 做 POC 與短期交付
- 先驗證內容編輯、發布、Visual Editing 與上架流程

### 如果 Readdy 後續補齊 CMS
- 就不必再重做一套相同的網站 CMS / 部署自動化
- 把產品投入改放到 CRM / OMO / 追蹤 / 歸因 / 客戶管理層

## 總結
Readdy 若補齊 CMS，代表它能涵蓋我們原本規劃中很大一段「網站內容管理與部署」能力。  
這不代表我們沒有產品空間，而是代表我們應該把差異化重心往「營運整合層」移動，而不是重複建一個網站 CMS 流程。
