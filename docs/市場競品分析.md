# 市場競品分析

更新日期：2026-03-09

## 1. 評估目標
- 能否快速以 AI 生成網站
- 能否讓客戶後續自行修改內容
- 能否整合進我們產品（白標 / API / SDK）
- 長期可控性（程式碼、CMS、SEO、OMO 整合）

## 2. 競品分析摘要

### Readdy
- 定位：AI 產碼工具（可下載 code，工程可接手）
- 優點：
  - 出稿快
  - code ownership 高（可自行部署、改造）
- 缺點：
  - 目前公開資訊下，embed/white-label 能力不明顯
  - 平台內完整 CMS/營運能力需自行補齊
- 適合情境：要做長期產品化、深度整合 CRM/OMO

### Wix
- 定位：一體化建站平台（AI + CMS + 發佈）
- 優點：
  - 非技術用戶上手快
  - 功能生態完整
- 缺點：
  - 平台綁定、客製上限
  - 常見負評集中在續約/價格與客服
- 適合情境：追求最快上線、低工程投入

### Webflow
- 定位：設計與 CMS 能力強的一體化平台
- 優點：
  - 設計自由度高
  - 專業團隊採用度高
- 缺點：
  - 學習曲線與成本爭議
- 適合情境：設計主導專案，且可接受平台限制

### Framer
- 定位：設計導向 AI 建站 + CMS
- 優點：
  - 視覺與互動呈現佳
  - 上線速度快
- 缺點：
  - 大型內容營運與複雜流程能力較有限
- 適合情境：品牌站、行銷頁快速交付

### 10Web
- 定位：WordPress 生態的 AI 建站工具
- 優點：
  - 快速生成 + WP 生態可延伸
  - 有白標方案可談
- 缺點：
  - 穩定度與支援品質評價有波動
- 適合情境：接受 WP 堆疊且希望快速上線

### Duda / Brizy / BaseKit / Site.pro（白標導向）
- 定位：提供 SaaS/代理商整合的 white-label website builder
- 優點：
  - 白標、API、SSO 能力成熟（尤其 Duda/Brizy）
- 缺點：
  - 會較綁平台架構
  - 與 Sanity 並行通常整合成本較高
- 適合情境：要在自家產品中直接提供建站器

### Gamma
- 定位：平台內 AI 生成與發佈（簡報/頁面體驗強）
- 優點：
  - 生成速度快、編輯體驗佳
  - 有 API（Pro 以上）
- 缺點：
  - 官方功能邊界：目前不支援完整 blog/e-commerce
  - Gamma site 不能嵌入其他網站
  - 網路評價兩極（Product Hunt 偏正向、Trustpilot 偏負向）
- 適合情境：快速內容型頁面，不適合深度嵌入式產品化建站流程

## 3. 綜合比較與建議

### 若目標是「最快讓客戶自己用」
- 優先：Wix / Duda / 10Web / Framer

### 若目標是「長期變成我們產品能力（可控、可整合 CRM/OMO）」
- 優先：Readdy + Sanity + Vercel + prerender（可加 BFF-lite）

### 若目標是「在我們產品內提供白標建站器」
- 優先評估：Duda / Brizy / BaseKit
- 不建議主打：Gamma（受嵌入限制）

## 4. 結論
- 短期效率：一體化平台通常更快上線
- 長期可控與產品化：Readdy 半自動化路線更符合我們方向

## 補充：BFF-Lite 說明

### 什麼是 BFF-Lite
BFF-Lite（Backend For Frontend - Lite）是介於前端與 CMS/第三方服務之間的一層「輕量後端」。  
它的目標不是一開始就做完整中台，而是先用最小成本解決安全、快取與資料格式統一問題。

### 為什麼需要 BFF-Lite
- 不讓前端直接暴露 CMS token 或敏感金鑰
- 統一前端資料格式，避免前端緊耦合 CMS schema
- 加入基本快取與錯誤降級，提升穩定性
- 後續若更換 CMS，前端改動可降到最小

### 最小可行範圍（MVP）
- 只做 read-only API（例如首頁、療程、醫師、文章列表）
- 提供基本快取與 revalidate 機制
- 做最小權限控制與日誌
- 不急著上複雜工作流（先求可用與穩定）

### 何時升級成完整 BFF
- 需要多資料源整合（CMS + CRM + 行銷/電商）
- 需要細緻權限、審核、版本控管與審計
- 需要進階觀測性（監控、告警、SLO）
- 需要高流量與多租戶治理能力
