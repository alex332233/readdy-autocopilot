# 部署前整理清單

更新日期：2026-03-21

本清單用於艾苜正式版網站在部署到正式環境前的最後整理，目標是避免因環境變數、CORS、內容 seed、legacy 頁面或壞圖連結等問題，導致正式站上線後才回頭修補。

### 前端
- 平台：Vercel
- 專案目錄：`amu-readdy-code-v2`
- Build command：`npm run build`
- Output directory：`out`

### Studio
- 平台：Sanity Hosted Studio
- 專案目錄：`amu-website-formal-studio`
- 部署指令：`npx sanity deploy`

### 資料層
- Sanity project：`pnfq5r7n`
- dataset：`production`

## 2. 正式環境變數

### 前端（Vercel）
正式環境至少確認以下 env：

```bash
VITE_SANITY_PROJECT_ID=pnfq5r7n
VITE_SANITY_DATASET=production
VITE_SANITY_API_VERSION=2025-01-01
VITE_SANITY_STUDIO_URL=<正式 Studio 網址>
```

補充：
- 正式站必須關閉 `VITE_ENABLE_VISUAL_EDITING`
- 若 preview 環境仍要保留視覺編輯，可在 preview env 另外設為 `true`

### Studio
正式 Studio 至少確認以下 env：

```bash

SANITY_STUDIO_DATASET=production
```

補充：

## 13. Vercel 第一次 CLI 部署的實際注意事項

這次已實際用 `npx vercel --yes` 驗證過第一輪部署流程，需注意以下幾點：

### 登入與權限
- 若本機既有 Vercel token 已失效，CLI 會直接報：
  - `The specified token is not valid. Use vercel login to generate a new token.`
- 這時需先執行：

```bash
npx vercel login
```

- 依裝置碼流程到瀏覽器授權後，再重新執行部署

### Preview 網址不是自動公開
- 這次 `npx vercel --yes` 成功後拿到的是：
  - `Preview URL`
- 若專案啟用了 Vercel Authentication / Deployment Protection，未登入 Vercel 的瀏覽器（例如無痕）會被導向 Vercel login
- 這代表：
  - CLI deploy 成功不等於外部客戶一定能直接開啟 preview 連結
  - 若要給外部人員預覽，需再確認 Vercel 專案的 deployment protection 設定

### Output directory 不能只靠自動偵測
- 這次 Vercel 雖然自動辨識為 Vite，但預設仍去找 `dist`
- 實際專案 build 輸出是 `out`
- 若未明確設定，會在部署時出現：
  - `No Output Directory named "dist" found after the Build completed`
- 這次處理方式：
  - 在前端專案新增 `vercel.json`
  - 明確指定：

```json
{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "framework": "vite",
  "buildCommand": "npm run build",
  "outputDirectory": "out"
}
```

### 先用 CLI 拿網址，再回填到 Studio 是可行的
- 這次已驗證：
  - `vercel` CLI 會在 stdout 直接輸出部署網址
- 因此可以先：
  1. 部署前端到 Vercel
  2. 取得 preview / production URL
  3. 再回填到 `SANITY_STUDIO_PREVIEW_ORIGIN`
- 這條流程後續可程式化，不一定要手動複製貼上
