# 專案交付軌

## 目標

以現有 `amu-readdy-code-v2` 為基礎，遷移到官方對齊的 React Router framework/SSR 架構，同時維持目前前端呈現、URL 與內容體驗不變。

## 核心原則

- 保留現有 UI component、頁面結構、CSS 與動畫。
- 把變更集中在資料層、preview/session 流程與 server-side 架構。
- 以「視覺不變、資料流重構」為遷移策略。

## 保留範圍

- `src/pages`
- `src/components`
- 現有 CSS、圖片、字型、動畫與互動呈現
- 現有 URL 設計與頁面資訊架構
- 現有 Sanity schema / seed / query 設計方向

## 重構範圍

- 目前以 browser 為主的 preview mode 流程
- `VisualEditing` 掛載條件與 preview 狀態判定方式
- draft 內容讀取方式
- server token、session 與 loader 架構
- Presentation 對應的 grants/live preview 管線

## 目標架構

### Preview / Visual Editing

- preview URL 經 `validatePreviewUrl()` 驗證
- preview 狀態寫入 cookie session
- route loader 讀 preview session
- `@sanity/react-loader` 建立 live preview
- `VisualEditing` 跟 session 狀態啟用
- viewer token 僅存在 server 端

### 內容資料流

- loader 決定 published / drafts
- component 吃 loader 提供的資料
- 前端不直接用 browser-side draft query 當正式方案

## 建議實作順序

1. 建立新的 React Router framework/SSR 骨架
2. 搬首頁資料流，不改 UI
3. 接 preview enable / disable / session / validatePreviewUrl
4. 接 `@sanity/react-loader` 與 `VisualEditing`
5. 驗證 Presentation 能正常連線
6. 逐頁搬移：
   - home
   - site settings / navbar / footer
   - about
   - insurance
   - team
   - cases
   - health education
   - featured treatments
7. 補 `resolve.locations` 與頁面文件映射
8. 做整站 smoke test

## 驗收標準

- 前端視覺與現站一致
- 正式站 Presentation 可正常連線
- draft / published 切換正確
- 點選頁面區塊可對應到文件
- 頁面 URL、SEO meta、內容輸出與現站一致
- Vercel / Sanity 環境設定可重複配置

## 主要風險

- `window` / `document` 依賴在 SSR 階段暴露問題
- hydration 與初始載入流程需要校正
- 目前 mock/default/fallback 與 seed 內容要嚴格對齊
- preview/session 流一次改動較大，驗證必須分段進行

## 建議節奏

- 先求首頁與 Presentation 打通
- 再逐頁搬移與驗證
- 不在搬移過程中同步重設計或重整 UI

## 預期成果

- 單一專案可穩定上線於 Vercel
- Sanity Presentation / Visual Editing 可在 local 與 production 使用
- 為後續 readdy CMS 模板提供第一手遷移樣板與驗證路徑
