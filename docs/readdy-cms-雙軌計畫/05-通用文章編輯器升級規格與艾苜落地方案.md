# 通用文章編輯器升級規格與艾苜落地方案

這份文件用來定義一套可重用的文章編輯器升級規格，目標是讓後續不同網站都能沿用同一套內容模型、前端 renderer 與遷移策略。

本文分成兩層：

- 通用規格：適合未來其他站共用
- 艾苜落地方案：本次如何套到 `衛教資訊` 與 `真實見證`

## 1. 目標

現有 Readdy 匯出站點的文章內容多半屬於：

- 固定欄位結構
- 簡單 heading + text + image
- 外部圖片 URL
- 不支援真正的 rich text 編輯

這會導致：

- 無法支援 H2 / H3、粗體、斜體、清單等常見文章需求
- 圖片只能貼 URL，不利於編輯與資產管理
- 前台 renderer 與 CMS schema 綁得太死
- 不同網站重複做類似改造

## 2. 第一版通用功能範圍

第一版通用文章編輯器支援以下能力：

- Paragraph
- H2
- H3
- Bold
- Italic
- Bullet list
- Numbered list
- Link
- Image upload to Sanity
- Image alt
- Image caption
- Blockquote
- Divider

第一版刻意不納入：

- H1
- 文字顏色
- 表格
- 影片 embed
- 自訂字級
- 複雜圖文混排 block

### 為什麼不開 H1

頁面本身通常已有主標題作為 H1。若內文再允許 H1，容易造成：

- SEO 結構混亂
- 編輯者濫用標題層級
- 前端 heading 階層失控

因此第一版建議內文只開 H2 / H3。

## 3. 通用內容模型建議

建議新增一個共用 block content schema，例如：

- `richArticleContent`

這個 schema 作為多個文章型 document 的共用欄位型別。

### 建議 block 類型

#### 文字 block

- styles
  - normal
  - h2
  - h3
- decorators
  - strong
  - em
- annotations
  - link

#### 清單

- bullet list
- numbered list

#### 圖片 block

- `image`
- fields
  - `alt`
  - `caption`

#### 其他 block

- `blockquote`
- `divider`

## 4. 通用 renderer 規則

前端 renderer 應統一處理以下邏輯：

- `normal` 對應一般段落
- `h2` / `h3` 依設計系統輸出正確 heading
- `strong` / `em` 用對應 tag render
- `bullet` / `number` 輸出為語意正確的 list
- `link` 支援外連與內部連結策略
- `image` 支援：
  - Sanity image URL
  - alt
  - caption
  - 合理尺寸與間距
- `blockquote` 有固定視覺樣式
- `divider` 有固定區隔樣式

## 5. 通用遷移策略

為了避免一次性重構打壞既有文章，建議採用雙軌遷移：

1. 新增新版欄位
2. 保留舊欄位
3. 前端優先 render 新版欄位
4. 舊文章暫時 fallback 舊欄位
5. 新增 / 編修的文章優先改用新版欄位
6. 最後視需要批次 migrate 舊資料
7. 全站確認穩定後，再移除舊欄位

### 不建議的做法

- 直接用新 schema 取代舊欄位
- 一次性刪除舊欄位
- 還沒做前台雙軌 render 就要求內容端全面改寫

## 6. 通用自動化切入點

這份規格後續很適合轉成自動化模板或腳本。可自動化的部分包括：

- 生成共用 block content schema
- 為指定 document 新增 `body` 欄位
- 自動建立 block renderer skeleton
- 自動建立新舊雙軌 render 邏輯
- 檢查 renderer 是否支援：
  - h2
  - h3
  - strong
  - em
  - list
  - image
  - alt
  - caption
  - blockquote
  - divider
- 檢查 document 是否仍依賴舊結構欄位
- 後續產生 migration checklist
- 產生文章 SEO 內容盤點報告，檢查 slug、SEO title、SEO description、封面 alt、摘要欄位

## 7. 艾苜網站落地方案

本次套用範圍為兩個內容模組：

- `衛教資訊`
  - 前端頁面：`/health-education`
  - document：`healthEducationArticle`
- `真實見證`
  - 前端頁面：`/cases`
  - document：`caseArticle`

兩者是不同頁面、不同 document，但內文編輯器能力可共用同一套 schema。

## 8. 艾苜第一版落地策略

### 衛教資訊

現況：

- 使用 `content[]`
- 每段只有 `heading + text + image(externalImage)`

規劃：

- 新增新版 `body` 欄位（共用 block content）
- 前端優先 render `body`
- 若 `body` 為空，再 fallback 舊 `content[]`

### 真實見證

現況：

- 使用固定欄位組合，如：
  - `before`
  - `after`
  - `conclusion`
  - `tips`
  - `medicalInfo`
  - `references`

規劃：

- 新增新版 `body` 欄位（共用 block content）
- 前端優先 render `body`
- 若 `body` 為空，再 fallback 既有欄位結構

## 9. 艾苜前端策略

建議前端拆成：

- 共用 block renderer
- `衛教資訊` 專用頁面組合
- `真實見證` 專用頁面組合

也就是：

- 內文 block renderer 共用
- 頁首、meta、FAQ、延伸資訊等區塊各自保留模組差異

## 10. 艾苜資料遷移建議

建議不要一開始就批次搬全部資料。

比較穩的順序：

1. 先建 schema
2. 先接前端雙軌 render
3. 先用 1 到 2 篇 `衛教資訊` 測試
4. 再用 1 到 2 篇 `真實見證` 測試
5. 確認內容端編輯體驗沒問題後，再決定是否 migrate 舊資料

## 11. 與既有 bug 的關係

先前已發現的問題：

- 刪除文章圖片後前台仍顯示殘留圖示

這個問題不建議在舊 `externalImage` 結構上再補 patch，而應在新版 block editor 導入時一起解掉。原因是：

- 新版圖片會改成 Sanity image asset
- renderer 也會改成新的 block 處理流程
- 舊問題自然會隨結構升級一起消失

## 12. 實作順序建議

1. 新增共用 block content schema
2. `healthEducationArticle` 新增新版 `body`
3. `caseArticle` 新增新版 `body`
4. 前端新增共用 renderer
5. `衛教資訊` 先接雙軌 render
6. `真實見證` 再接雙軌 render
7. 驗證內容端編輯流程
8. 最後再考慮 migration 與清理舊欄位

## 12.1 第一階段實作完成狀態

目前第一階段已完成：

- Studio 已新增共用 `richArticleContent`
- `衛教資訊` 已新增 `body`
- `真實見證` 已新增 `body`
- 前端已建立共用 `RichArticleRenderer`
- `衛教資訊` 已完成 `body` 優先 render + 舊 `content` fallback
- `真實見證` 已完成 `body` 優先 render + 舊欄位 fallback
- Presentation / preview 已可直接看到新版文章內文的即時變化

目前尚未完成：

- 舊文章批次 migrate
- 何時關閉舊欄位輸入流程
- blockquote 進階編輯 UX（如同一引用內的特殊換行需求）

## 12.2 這次實作對未來自動化最重要的經驗

### A. 先做 schema，再做單一內容類型，再做第二個內容類型

不要一開始就同時重構：

- Studio schema
- `衛教資訊`
- `真實見證`
- 舊資料 migrate

比較穩的順序是：

1. 先建共用 block schema
2. 先打通第一個內容模組
3. 驗證 preview / visual editing
4. 再複用到第二個內容模組

這很適合之後自動化成階段式腳手架。

### B. 前端 renderer 要先做最小可用版本，不要第一版就追求完整套件化

這次沒有先引入額外 rich text 套件，而是先做最小可用 renderer，直接驗證：

- schema 是否合理
- preview 是否會動
- visual editing 是否可用

未來自動化工具可以先產生：

- 最小 block renderer
- 再由專案視需求替換成更完整套件

### C. 雙軌 render 是必要策略，不是可選優化

如果現場已有舊資料，直接切換新欄位風險太高。

這次已證明比較穩的模式是：

- `body` 存在就 render `body`
- 否則 fallback 舊欄位

這條規則應該成為未來文章編輯器自動化模板的預設值。

### D. 圖片一定要直接走 Sanity image asset

舊的外部圖片 URL 結構會帶來：

- 編輯體驗差
- 刪除殘留 bug
- preview / visual editing 對應不穩

所以未來自動化若偵測到文章編輯器升級，應預設：

- 使用 Sanity image block
- 支援 `alt`
- 支援 `caption`

### E. 文章主體用 block editor，結構欄位獨立保留

這次驗證後，合理的拆法是：

- block editor 管文章主體
- 獨立欄位保留：
  - title
  - summary
  - coverImage
  - faq
  - seo
  - 某些明確資訊盒

這點很適合作為未來生成 schema 的預設規則。

### F. 新增文章 UX 必須跟新版 editor 一起做

這次實作後確認，若只新增 `body` block editor，但沒有同步調整新增文章流程，客戶仍可能被舊欄位卡住。

本次後續補上的規則：

- `衛教資訊` 新增文章時，自動帶入 `publishDate` / `updatedDate`
- `真實見證` 新增文章時，自動帶入 `publishDate`
- `tags` / `body` / `references` 等陣列欄位預設為空陣列
- 衛教舊版 `content` 改 optional

未來自動化模板應把這些列為同一組交付，而不是只交付 schema 和 renderer。

### G. 文章 SEO 欄位與 slug 要先支援雙軌

文章改成 slug URL 時，不應立刻移除舊數字 id URL。

比較安全的策略是：

- `slug` 欄位先加入 schema
- 列表頁優先連到 slug
- detail loader 同時接受 slug 與舊 id
- seed script 只補缺漏 slug
- 上線前再規劃正式 slug 與 301

這可以避免 SEO 改造過程中讓舊連結直接失效。

### H. 日期欄位應標準化成 date

文章 `publishDate` / `updatedDate` 應使用 Sanity `date` 型別，而不是 `string`。

原因：

- Studio 會自動提供日期選擇器
- 格式固定為 `YYYY-MM-DD`
- 未來接 SEO structured data 較穩
- 客戶不會輸入不一致格式

若是從 Readdy 或舊資料轉入，應額外準備 normalize script，把舊格式日期轉成 `YYYY-MM-DD`。

### I. slug seed 只是過渡，不代表正式 SEO 完成

自動 seed 出來的 `health-education-1` / `case-1` 這種 slug，只能讓系統先有穩定 URL。

正式 SEO 仍需要人工或策略規劃：

- 簡短英文
- 拼音
- 服務關鍵字
- 避免純數字或無意義字串

因此未來自動化工具應把 slug seed 標記為「補洞」，不要把它標記為「SEO 已完成」。

### J. 先做 SEO 盤點工具，再決定是否批次修改資料

SEO 欄位清理不建議一開始就寫 patch script 直接改資料。

比較穩的順序是：

1. 先用 read-only audit script 列出問題
2. 標記過渡 slug、缺 SEO title、缺 SEO description、缺封面 alt
3. 讓內容端決定哪些要人工改、哪些可批次補
4. 最後才寫 patch script

本次已新增 `audit:article-seo`，可作為未來其他站的自動化雛形。

盤點工具預設只檢查已發布文章，GROQ 需要排除 `drafts.**`：

```groq
*[_type == $type && !(_id in path("drafts.**"))]
```

否則同一篇文章只要有 draft，就會被 published + draft 算成兩篇，導致總數與重複清單失真。工具也應另外檢查「已發布文章本身是否有重複文章編號」，這才是真正需要修資料的重複問題。

## 13. 後續可擴充功能

如果未來有更多需求，可考慮第二版再加入：

- 表格
- 影片 embed
- callout / 提醒區塊
- FAQ inline block
- 圖文雙欄 block
- 自訂引用樣式

這些不建議放入第一版，以免初次落地複雜度過高。
