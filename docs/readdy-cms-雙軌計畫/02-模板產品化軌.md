# 模板產品化軌

## 目標

從原始 readdy 輸出沉澱一份可重複套用的 CMS-ready starter，讓後續網站更容易完成：

- Vercel 上線
- Sanity Studio 建置
- seed 初始內容
- Presentation / Visual Editing 啟用

## 核心判斷

如果目標是長期產品化，而不是只修單一專案，建議模板主路線以 React Router framework/SSR 為基礎，而不是在純 Vite SPA 上持續補 bridge。

## 模板定位

建議將 readdy 未來輸出分為兩類：

### 1. Lite brochure starter

適合：

- 純展示站
- 無 CMS
- 只需要 published content

特性：

- 保留 `Vite + React + TypeScript`
- 上線快
- 不承諾完整 Presentation / Visual Editing

### 2. CMS-ready starter

適合：

- 要接 Sanity
- 要上 Presentation / Visual Editing
- 要有可維護 preview 與內容工作流

特性：

- React Router framework/SSR
- 內建 session / preview / loader / token 邏輯
- 更適合做自動化生成

## CMS-ready starter 建議標配

- React Router framework/SSR 基礎骨架
- `src/sanity` 標準目錄
- Sanity Studio scaffold
- preview secret / validate route
- cookie session utilities
- `@sanity/react-loader`
- viewer token env 規範
- `VisualEditing` root wiring
- `resolve.locations`
- seed scripts
- Vercel / Sanity env 範本

## 可自動化生成的項目

- Studio schema 基礎文件
- query / type / defaults 基礎檔
- preview routes / session utilities
- env.example
- seed 檔骨架
- Vercel / Sanity deploy checklist
- 頁面對應的文件 resolver 骨架

## 建議先不要全自動的項目

- 複雜 detail page 的 document mapping
- 非典型內容模型
- 特殊 SEO / schema.org 邏輯
- 高度客製的導覽或站內互動規則

## 產品化里程碑

### 第 1 階段：整理模板規格

- 從原始 readdy 輸出建立乾淨的 CMS-ready starter
- 把必要資料夾、檔案與 env 規範固定下來

### 第 2 階段：半自動化

- 將 schema / seed / query / types / preview routes 生成半自動化
- 減少每案人工複製與手改

### 第 3 階段：整合進 readdy 產出流程

- 在生成網站時提供 CMS-ready 選項
- 讓使用者可選擇 Lite 或 CMS-ready 輸出

## 與本案的關係

本案不建議退回原始輸出重做整站；本案更適合從現狀遷移完成交付。

但本案的遷移經驗非常適合作為產品化參考，後續應另從原始 `readdy-code` 做一份乾淨版 starter，避免把單案歷史包袱直接帶進模板。

## 預期成果

- 建立可複用的 CMS-ready starter
- 縮短新專案從 readdy 輸出到 Vercel + Sanity 上線的時間
- 降低每次接入 Presentation / Visual Editing 的不確定性
- 讓內容工作流更接近官方推薦做法，後續維護更穩定
