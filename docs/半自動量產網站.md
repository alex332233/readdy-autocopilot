# 半自動量產網站評估小結

## 評估場景
我們正在評估一條「半自動量產網站」流程：  
以 Readdy 作為前端產碼起點，後續接上 Sanity（可視化內容編輯）與 Vercel（部署與發佈），目標是讓設計師盡量自助完成上架，降低每次都要工程師介入的成本。

## 結論
這條路值得做，但目標應設定為「高自助率（80-90%）」而非「零工程」。  
原因是新元件第一次仍需要工程師建立規則；一旦納入標準元件庫後，設計師即可重複使用並自主上架。

## 怎麼做
1. 先建立一次共用地基
- 前端殼層（共用 layout、SEO、路由規範）
- Sanity 整合層（schema、visual editing、preview/publish）
- Vercel pipeline（build + prerender + deploy）

2. 建立標準元件庫
- 先定義 8-12 個高頻區塊（Hero、服務卡、FAQ、輪播、部落格列表等）
- 每個元件綁定可編輯欄位（文字/圖片/連結/排序/開關）

3. 設計師日常流程
- 使用既有元件做內容映射與小量校正
- 在 Sanity Visual Editing 檢視與發佈
- 多數情境不需修改程式碼

4. 例外處理
- 遇到新型元件時，工程師介入一次（元件 + schema + mapping）
- 完成後納入元件庫，供後續專案重複使用

## 建議評估方式
1. 先做 1 個 MVP（2-4 週）並用 1 個真實案驗證
2. 追蹤三個核心指標
- 設計師自助率（目標 >70%）
- 新站交付時間是否明顯下降
- 工程師介入次數是否隨案量下降
3. 若達標，擴大元件庫與自動化深度；若未達標，調整元件標準化範圍與流程
